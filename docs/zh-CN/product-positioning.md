# 产品定位

Repo as Review OS 是一个围绕 GitHub 构建的 AI 原生复习系统。

它不是某一个 AI 产品的专属项目，不是 Obsidian 专属工作流，也不是多 Agent 框架。

它的默认模型是：一个主 AI 工具，加一个 GitHub 仓库。

## 一句话

用 GitHub 承载复习系统，用一个有能力的 AI 工具维护它。

## 它靠什么运行

关键不在品牌，而在能力。AI 需要能够：

1. 读取仓库文件；
2. 创建和修改文件；
3. 遵守项目规则；
4. 记住用户偏好；
5. 理解权限边界；
6. 汇报每次改动。

## 为什么最终不以 Obsidian 为中心

很多用户会从 Obsidian 开始，因为它像知识库。

一个实际演化路径可能是：

```text
Obsidian
→ Obsidian + Git
→ Obsidian + Git + GitHub
→ GitHub + 一个 AI 工具
```

最后一种形态更轻：

- 核心工作流不再依赖额外本地软件；
- 仓库本身就是共享事实源；
- GitHub 自带版本历史；
- AI 工具可以直接读取和修改同一套结构；
- 用户只需要一个 AI App 加一个仓库连接器就能维护系统。

Obsidian 仍然可以作为个人编辑器，但不再是系统中心。

## 核心运行环境

最小运行环境：

```text
一个 AI 工具 + 一个 GitHub 仓库 + 一个规则文件
```

最佳运行环境：

```text
一个主 AI 工具
+ GitHub 或 MCP 连接器
+ 记忆或项目指令
+ 单仓库写入权限
+ 清楚的安全边界
```

## 什么放到网上

GitHub 适合保存：

- 结构；
- 模板；
- 控制台；
- 索引；
- 来源状态说明；
- 清理后的示例；
- 接手规则；
- 网站文案。

GitHub 是操作层。

## 什么留在本地

本地适合保存：

- 原教材；
- 有版权的 PDF；
- 原始扫描件；
- 私人截图；
- 老师文件；
- 大型媒体；
- 用户不想上传的内容。

本地是受保护的来源层。

## AI 如何处理本地文件

AI 不能假装自己读过本地文件。

如果某个来源只在本地，AI 应该先在 GitHub 里创建来源记录；只有必要时，才让用户提供最小摘录或截图。

## 权限哲学

最好的默认状态是：只给一个仓库写入权限。

这样 AI 可以在 Review OS 仓库里顺畅工作，同时高风险操作仍然需要用户确认。

可以默认执行：创建文档、更新模板、更新控制台、标记 `todo`、添加清理后的示例、汇报改动。

必须先问用户：删除文件、改变公开状态、上传本地私人文件、发布真实学习记录、接触密钥、修改许可证。

## 最终承诺

Repo as Review OS 让用户用下面这套东西运行复习系统：

```text
一个 AI App
一个 GitHub 仓库
一个连接器
一层记忆或项目规则
```

它把复杂工具链收束成一个 AI 能理解、能维护、能持续接手的仓库。
