# 产品定位

GitLearnOS 是一套轻量、用户拥有的学习控制层：一个有能力、可替换的主 AI Agent，加一个 GitHub 仓库。

它自动整理任意渠道的学习输入，生成针对性问题，并按用户政策把长期状态写回。它不是完整教学平台，不绑定 ChatGPT，不要求 Obsidian，也不是多 Agent 框架。

## 一句话

```text
整理任何地方发生的学习，生成下一组真正有用的问题，并持续同步用户拥有的状态。
```

## 产品判断

真实学习本来就分布在老师、课堂、补习、纸笔、教材、题库、同学与 AI 中。轻量系统应该协调这些渠道，而不是把它们强行搬进自己的平台。

因此，GitLearnOS 把：

- 一个仓库作为学习者拥有的控制面；
- 根目录文件作为共享政策与跨学科协调；
- `subjects/<subject>/` 作为聚焦的学科状态；
- 当前 AI 作为可替换的执行者；
- 真人老师与外部平台作为一等学习渠道；
- 整理与出题作为核心能力；
- AI 实时教学作为可选路径。

在校与自学共享学习者层偏好，但保留不同目标、期限、方法和证据。补弱、考试、探索、超前学习、项目、研究和技能都使用同一套控制层，只改变任务形式与验收标准。

## 三项核心能力

| 能力 | 产品承诺 |
|---|---|
| 整理 | 捕获、判科、分类、关联、去重、同步反馈并给出下一步 |
| 出题 | 生成有来源的诊断、练习、变式、迁移、复习和外部答疑问题 |
| 自动化 | 透明、幂等、可撤销地执行安全写回与调度 |

讲解、搜索、评分、间隔复习与学习者建模服务于这三项能力。

## 轻量实现

| 学习需要 | GitLearnOS 实现 |
|---|---|
| 跨渠道捕获 | 学科 inbox、来源记录、活动记录 |
| 聚焦上下文 | 一个仓库中的 `subjects/<subject>/` 文件夹 |
| 用户掌控 | 根目录学习政策、Markdown、Git 历史、撤销 |
| 自动整理 | 学科路由、唯一状态归属、dashboard |
| 个性化出题 | 出题 Skill、复习记录、外部问题包 |
| 老师协作 | 缺口生命周期、问题包、反馈同步 |
| 掌握证据 | 学习者结果、提示程度、延迟与迁移检验 |
| 自动化 | 即时与接手规则；可选真实调度器 |
| 扩展 | 一个路由入口和聚焦的 `SKILL.md` 工作流 |

## 为什么用 GitHub

GitHub 提供稳定路径、明确归属、版本历史、可检查证据、跨 AI Runtime 迁移能力，以及 Agent 出错后的恢复能力。

仓库保存被动状态与声明式政策。当前 Agent 负责整理、出题、按需教学和写回。后台定时只是可选、依赖 Runtime 的能力。

## 为什么是一个仓库加学科文件夹

一个仓库让授权、备份、政策和用户掌控保持简单。学科文件夹减少无关上下文，避免数学、语言、编程等状态混成一堆。

Agent 自动判科，只有可能写错时才询问，并接受自然语言纠正。它创建当前活跃学科和第一个有用文件，而不是空目录体系。

详见[学科文件夹模型](subject-folder-model.md)。

## 为什么只需要一个主 Agent

现代工具型 Runtime 可以在一个工作区内读取状态、推理、执行和验证。GitLearnOS 以一个主 Agent 按需加载一个工作流为标准。

强制多 Agent 会增加交接成本，不是轻量产品的必要条件。

## 适用用户

GitLearnOS 适合：

- 同时在老师、学校、自学、纸笔、平台和 AI 中学习；
- 希望 AI 减少整理与复习维护成本；
- 希望题目基于个人状态，而不是随机题单；
- 重视可读、可迁移、自己拥有的记录；
- 不想再部署一个完整平台。

## 产品边界

最小运行条件：

```text
一个有能力的主 Agent
+ 一个私有仓库与一个活跃学科文件夹
+ GitLearnOS 规则
+ 一个学习目标或输入
```

连接文件、网页研究、代码执行、可视化、原生记忆和真实调度器都可以增强体验，但不是基础控制层的前提。

## 最终承诺

成功不是仓库更大、文件更多或与 AI 聊得更久。成功是学习者只报告一次事件，就获得有用的整理或问题，并能相信状态被正确判科、更新、展示、迁移和撤销。
