# 权限与边界模型

本页解释 Repo as Review OS 的权限边界。

运行路径也请参考 [运行环境自适应](runtime-self-adaptation.md) 和 [本地运行说明](../local-runtime-note.md)。

Repo as Review OS 一开始就应该问一个问题：

**AI 到底可以在不反复询问用户的情况下读取和修改什么？**

如果 AI 不清楚自己的权限边界，这个系统就会变得很难用。

## 1. 权限等级

### Level 0：没有仓库权限

AI 只能解释理念和给出操作说明，不能检查或修改文件。

### Level 1：只读仓库权限

AI 可以检查仓库并提出修改建议，但不能写入文件。

### Level 2：单个目标仓库读写权限

AI 可以在一个指定仓库里创建和修改文件。这是推荐默认状态。

### Level 3：广泛账号权限

AI 可以访问多个仓库或其他连接服务。能力更强，但风险也更高。除非确实有必要，否则不要默认使用。

### Level 4：只有本地工作区权限

AI 可以读取和写入本地文件，但这不等于它可以更新 GitHub。

除非它已经验证过，否则不能声称已经完成 GitHub 部署、push、Issues 或远程仓库更新。

## 2. 推荐默认状态

使用 Level 2：

- 只授权一个目标仓库；
- 允许读取和写入；
- 默认不给整个账号权限；
- 不接触密钥；
- 不执行破坏性操作。

这能让 AI 足够顺手地工作，同时边界清楚。

## 3. 降低操作摩擦

为了减少每次都点允许，用户应该：

1. 创建或选择一个目标 Review OS 仓库；
2. 只把这个仓库授权给 AI 工具；
3. 允许 AI 在这个仓库里正常创建和修改文件；
4. 要求 AI 每次修改后汇报；
5. 保持删除、公开、密钥相关操作需要确认。

## 4. AI 可以默认执行的操作

如果 AI 拥有这个仓库的读写权限，它可以默认执行：

- 创建缺失文档；
- 创建模板；
- 更新索引；
- 把不完整文件标记为 `todo`；
- 添加清理后的示例；
- 更新控制台；
- 记录改动。

## 5. 必须确认的操作

AI 必须先询问用户，才能：

- 删除文件；
- 改变仓库公开状态；
- force push；
- 上传本地私人文件；
- 发布真实复习记录；
- 修改许可证；
- 接触 token、密码、cookie、密钥。

## 6. 运行诚实规则

AI 必须说清楚自己实际上能做什么。

除非它能在仓库中验证，否则不要说某个 GitHub 动作已经发生。

可以这样说：

```text
我可以编辑本地文件，但这里没有 GitHub 写入权限。
```

```text
我可以读取这个仓库，但需要你授权写入后才能更新文件。
```

```text
我已经在目标仓库中更新了这些文件：...
```

## 7. 单一 AI 模式

本项目默认采用一个主 AI，例如 ChatGPT，作为维护者。

不要求多 AI 工作流。其他工具可以存在，但主模型应该能读取仓库、判断任务、修改文件并汇报结果。

## 8. 什么不能完全无感

有些操作永远需要用户确认，因为平台必须保护用户：

- 第一次连接 GitHub；
- 授权仓库；
- 高风险写入；
- 读取本地文件；
- 发布内容或改变可见性。

目标不是完全没有授权提示，而是设定清楚的单仓库边界，让日常维护不再反复打断。
