# 自动化模型

Repo as Review OS 的自动化必须先说明当前定时环境到底能做什么。

定时提醒不一定等于仓库工人。

运行环境能力对照见 `docs/zh-CN/automation-runtime-matrix.md`。

## 两层自动化

```text
提示词型自动化
→ 生成接手卡
→ 提醒用户下一步做什么
→ 不声称仓库工作已经完成

工人型自动化
→ 拥有仓库或工具权限
→ 读取仓库状态
→ 把输出写回仓库
→ 汇报真实改动
```

## 重要边界

有些聊天软件里的自动化只能按时间运行一段提示词。

这种情况下，它只能算点火器，不能算最终执行者。

除非当前运行环境真的拥有对应工具，否则自动化不能声称已经读取、检查、修复、更新或导出了仓库内容。

规则很简单：

```text
没有工具权限 → 只能生成接手卡。
有工具权限 → 才能真正执行仓库工作。
```

## ChatGPT 风格的提示词型自动化

当定时自动化不一定拥有实时仓库工具时，使用这种模式。

它应该输出一张简短接手卡，包括：

1. 当前定位：这只是接手卡 / 点火器。
2. 当前对话或 agent 接下来要继续做什么。
3. 应该优先检查的仓库范围。
4. 需要检查的风险，例如来源缺失、控制台过期、复习项未回炉。
5. 用户继续执行的口令，例如：继续检查仓库。

最后必须有清楚提醒：

```text
本自动化本身不代表仓库任务已经完成。需要在有工具权限的对话或 agent 中继续执行。
```

## 仓库整理器

目的：让仓库保持可用。

如果作为工人型自动化运行：

1. 读取控制台和索引；
2. 找出不完整文件；
3. 找出过期 TODO；
4. 更新控制台；
5. 汇报改动。

输出可以写入：

```text
automations/repository-organizer/YYYY-MM-DD.md
```

如果只是提示词型自动化，就只输出接手卡。

## 题单生成器

目的：根据仓库历史生成复习任务。

如果作为工人型自动化运行：

1. 读取最近题目卡；
2. 读取旧薄弱点；
3. 读取活跃模型；
4. 检查哪些内容最近没有出现；
5. 生成小型复习题单；
6. 写回仓库。

输出可以写入：

```text
automations/practice-generator/YYYY-MM-DD.md
```

如果只是提示词型自动化，就输出接手卡，并等待用户在有工具权限的当前对话中继续。

## 接手卡模板

提示词型自动化可以使用这种结构：

```text
标题：准备仓库接手卡

这只是定时点火器和上下文交接，不是真正的仓库工人。

当前对话实际使用仓库工具之前，不要声称仓库工作已经完成。

输出简短接手卡：
1. 当前定位。
2. 当前对话下一步任务。
3. 优先检查的仓库范围。
4. 风险或缺失来源。
5. 用户继续执行口令。
```

## 出题接手卡模板

```text
标题：准备出题接手卡

这只是定时点火器和上下文交接，不是真正的题单生成器。

当前对话实际读取仓库之前，不要生成正式复习题单。

输出简短接手卡：
1. 当前定位。
2. 仓库读取前禁止声称的内容。
3. 用户继续执行口令。
4. 优先读取的仓库范围。
5. 仓库读取后的出题规则。
```

## 可选工人实现

脚本或 GitHub Actions 工人以后可以加，但不是协议运行的前提。

如果要加，应该从很小的检查开始：

```text
检查必要文件
检查必要目录
写一个简单健康报告
避免自动重写学习内容
```

这种工人是验证层，不是整个学习 agent。

## Git 历史的作用

Git 历史帮助 AI 判断新旧。

AI 可以根据文件更新、提交历史和控制台记录判断什么是新的、什么很久没复习、什么应该回炉。

但这只在当前环境真的能读取 Git 历史时成立。

## 边界

定时任务不能假装使用了实际不可用的工具。

如果环境只是提示词型，自动化就只是点火器。

如果环境拥有仓库权限，自动化才可以作为工人执行。

删除文件、公开私人记录、上传本地材料之前仍然必须先确认。
