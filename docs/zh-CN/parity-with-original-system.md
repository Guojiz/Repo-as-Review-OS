# 与原系统的对齐分析

本文检查 GitLearnOS 能否复刻原 `zhongkao` 仓库系统的能力。

原始 `zhongkao` 系统是这个项目最重要的现实基准。外部辅导研究可以作为设计参考，但第一标准是：已经真实跑通过的学习闭环，能不能被做成更轻、更通用、方便其他 AI 工具部署的模板。

仓库名仍然是 `Repo-as-Review-OS`，用于保持链接连续。

## 可以复刻什么

### 1. GitHub 作为主操作层

可以。新模板可以把 GitHub 当作主地图、状态库、模板库和接手界面。

### 2. AI 主维护

可以。旧系统后期已经转向 ChatGPT 主维护。GitLearnOS 通过 `START-HERE.zh-CN.md`、`AGENTS.zh-CN.md` 和接手协议把这一点明确写出来。

### 3. 内容完整性

可以。旧系统强调来源、摘要、索引和不完整文件之间的区别。GitLearnOS 用来源规则、内容标签和本地来源记录来复刻这一点。

### 4. 模板和控制台

可以。题目卡、写作素材卡、每日复盘、知识缺口记录和控制台都可以复刻。

### 5. 复习与下一步行动闭环

可以。真正有用的是这条闭环：

```text
目标
→ 来源
→ 拆解
→ 模型
→ 知识缺口
→ 题单
→ 控制台
→ 下一步行动
```

## 不能只靠 GitHub 完全复刻什么

### 1. 完全无感的本地检查

如果 AI 无法访问本地文件，它就不能自己检查教材、截图、PDF 或私人笔记。

### 2. 自动后台执行

聊天软件里的单一 AI 通常不能永远在后台运行。它可以在会话中维护仓库，但定时任务和后台任务取决于平台能力。

### 3. 权限弹窗

AI 无法消灭所有权限提示。第一次连接、授权仓库、读取本地文件、删除、发布内容，仍然需要用户确认。

## 新推荐模型

新模板不要求多 Agent 系统。

推荐模型：

```text
ChatGPT 或一个主 AI
→ GitHub 仓库
→ 只有需要时，用户才提供本地摘录
```

## 与旧系统的区别

旧系统尝试过本地 Agent、Obsidian、Git、GitHub 和自动巡检。

GitLearnOS 更轻：

- 一个主 AI；
- 一个 GitHub 仓库；
- 清楚的权限边界；
- 本地保存受保护的原始资料；
- GitHub 作为共享操作层；
- ChatGPT 记忆或项目规则作为个人偏好层。

## 实际结论

GitLearnOS 可以复刻旧系统的结构和方法。

但如果不增加本地 Agent 或本地文件权限，它不能完美复刻无感本地自动化。

因此，默认产品应该承诺的是：

**一个轻量的 GitHub + AI 学习系统：一个仓库、一个主 AI，本地资料只在需要时由用户提供。**
