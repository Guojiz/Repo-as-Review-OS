# 运行环境自适应

Repo as Review OS 不应该假设每个 AI 工具的用法都一样。

仓库提供协议。当前 AI 环境决定这个协议应该怎么部署。

## 核心规则

```text
先识别运行环境。
再根据运行环境调整部署路径。
```

## AI 开始前要先判断什么

在帮用户搭建仓库前，AI 应该先回答：

```text
我现在运行在哪个环境里？
我能不能读取目标仓库？
我能不能写入目标仓库？
我支不支持 skill？
我是不是要改用 memory 或 project instructions？
我能不能使用本地工作区？
哪些步骤必须由用户手动完成？
```

## 需要时查看当前产品说明

如果部署依赖某个 AI 软件的 app、connector、project、workspace 或权限流程，AI 不应该瞎猜。

它应该在可用时查看当前产品说明，或者让用户完成必要的手动步骤。

AI 应该明确说：

```text
我不能在这里替你完成这一步。请你手动完成后告诉我。
```

## 不同环境的部署方式

不同环境可以用不同路径：

```text
ChatGPT Project
→ 项目指令、记忆回退、可选 GitHub connector、本地科目文件夹

Codex
→ 仓库搭建、文件修改、模板复制、结构化交接

Claude Code / Cloud
→ 如果有工作区权限，就按仓库协议执行

Cursor / Windsurf
→ 本地工作区编辑和仓库维护

MCP worker
→ 明确工具权限下执行

GitHub Actions
→ 确定性检查和简单报告
```

## 部署时的回复格式

AI 开始部署时，应该先输出：

```text
已识别运行环境：
- 当前环境：
- 读取权限：
- 写入权限：
- 是否支持 skill：
- memory / project instructions 回退：
- 需要用户手动完成的步骤：

部署计划：
1. 确认目标仓库
2. 检查是空仓库还是已有文件
3. 选择 skill 或记忆回退
4. 创建或确认结构
5. 询问第一个学习目标
6. 汇报改动
```

## 原则

仓库说明是地图。

当前 AI 运行环境是交通工具。

AI 应该保留 Repo as Review OS 的协议，同时根据自己的工具能力调整真实部署路径。
