# 运行环境自适应

GitLearnOS 规定学习协议，当前 AI 环境提供工具能力。

## 检测能力，不只看产品名称

行动前判断：

```text
能否读取模板仓库？
能否读取并写入目标仓库？
能否访问学习者授权的来源文件？
能否加载文件型 Skills 或仓库指令？
是否有原生记忆或项目指令？
能否真正创建自动化，还是只能写提示词？
哪些步骤确实必须由用户完成？
```

安全时通过现有工具测试能力。产品名称本身不能证明权限。

## Work 模式快速路径

当当前工作区能检查文件、调用连接应用或仓库工具，并写入目标仓库时使用；正确连接的 ChatGPT Work 会话就是一种例子。

```text
验证目标与权限
→ 检查已有目标状态
→ 只读入口规则与活跃文件
→ 直接完成学习任务
→ 写回长期证据
→ 验证改动文件
→ 汇报结果
```

当前获授权环境自己能完成时，不要再要求用户粘贴仓库内容、重建文件或手工执行连接操作。

连接文件或应用可以作为当前来源界面。GitHub 保存长期结构化学习状态，不要把所有原件都倾倒进去。

## 可操作仓库的 Agent 路径

Codex、Claude Code、Cursor 或其他在本地 checkout 中工作的 Agent 应：

- 遵守仓库级指令；
- 修改前检查工作区；
- 保留无关改动；
- 使用同样的学习会话与证据规则；
- 将本地原件保留在公开 GitHub 之外；
- 汇报验证与改动文件。

如果它就是用户选择的主 Agent，它可以直接运行 GitLearnOS，不只用于部署其他 Runtime。

## 原生聊天路径

如果 AI 有记忆或项目上下文，却没有目标仓写入权限：

- 使用当前对话进行实际教学；
- 只有确实可用时才读取连接来源；
- 输出简短的待写回区块；
- 明确标记仓库尚未更新；
- 将稳定偏好与快速变化的学习状态分开。

## 单上下文回退

只请求：

- 活跃目标；
- 相关学习画像摘录；
- 一份来源、模型、缺口或复习摘录；
- 当前任务。

完成一个聚焦的学习闭环，并返回之后可精确写入的状态变化。不得假装拥有长期记忆或仓库权限。

## 自动化诚实性

必须区分：

```text
真实定时自动化
→ 已通过可用调度系统创建

提示词或计划
→ 只有说明，不会自动运行
```

自动化输出真正存在后才能记录。基础 GitLearnOS 闭环不能依赖后台自动化。

## 权限边界

已获授权时，可以完成用户任务需要的普通目标仓写入。删除、大范围覆盖、公开状态变化、发布私人内容、接触密钥或更换许可证前必须询问。

## 规则

运行机制可以适配，学习标准不能降低。所有环境都应保持来源诚实、学习者先尝试、基于证据评分、选择性写回和清楚的下一步行动。
