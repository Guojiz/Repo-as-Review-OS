# 自动化运行环境矩阵

Repo as Review OS 不要求绑定某一种固定自动化引擎。

自动化是一层执行接口，不同 AI 环境可以用不同方式实现它。

## 核心规则

```text
同一套仓库协议。
不同的运行能力。
```

仓库应该定义：

- 目标放在哪里；
- 来源放在哪里；
- 模型放在哪里；
- 复习放在哪里；
- 自动化输出应该放在哪里；
- 没有工具权限时 AI 不能声称什么。

具体能执行到哪一步，由当前运行环境决定。

## 运行环境矩阵

| 运行环境 | 最适合的角色 | 能否读写仓库 | 推荐用途 |
|---|---|---:|---|
| ChatGPT 定时自动化 | 提示词接手卡 | 通常不能 | 发送点火卡，让用户在有工具权限的当前对话继续。 |
| 带 GitHub connector 的 ChatGPT | 交互式工人 | 连接后可以 | 在当前对话中读取仓库、修改文件、汇报改动。 |
| Codex | 仓库工人 | 授权后可以 | 搭结构、改文件、运行代码、执行明确仓库任务。 |
| Claude Code / Cloud | 仓库工人 | 取决于配置 | 适合较长的编码、维护和仓库接手任务。 |
| Cursor / Windsurf | IDE 工人 | 在工作区内可以 | 本地编辑文件、套模板、维护学习仓库。 |
| MCP worker | 工具工人 | 配好后可以 | 作为真正的自动化工人执行有权限的工具任务。 |
| GitHub Actions | 脚本工人 | 可以，但偏脚本 | 做确定性检查、生成简单报告、验证仓库结构。 |
| 手动 agent prompt | 人类触发工人 | 取决于当前工具 | 用户把接手提示词交给当前有权限的 AI。 |

## 不能默认假设什么

不要默认每个 AI 工具都能：

- 访问 GitHub；
- 修改文件；
- 运行脚本；
- 跨运行保持状态；
- 检查只在本地的材料；
- 在没有读取仓库前生成正式复习题单。

## 提示词型自动化

当自动化环境只是定时提示词时，使用这种模式。

它应该输出：

```text
接手卡
下一步口令
优先检查的仓库范围
需要验证的风险
明确说明仓库任务尚未完成
```

它不应该直接生成最终仓库输出。

## 工人型自动化

当环境拥有仓库或工具权限时，使用这种模式。

它可以：

```text
读取 dashboard
读取当前目标
读取来源记录
读取模型
生成复习输出
更新 dashboard
写入自动化报告
汇报改动文件
```

## 可选 GitHub Actions 层

GitHub Actions 有用，但不应该成为必需项。

轻量 GitHub Actions 工人可以检查：

```text
dashboard.md 是否存在
goals/main-goal.md 是否存在
sources/ 是否存在
models/ 是否存在
reviews/ 是否存在
templates/ 是否存在
automations/ 是否存在
```

这只是结构检查，不是 AI 学习 agent 的替代品。

## 定位

Repo as Review OS 首先是一套可迁移的学习仓库协议。

它可以支持脚本、GitHub Actions 和定时工人，但不应该被锁死到某一种自动化引擎里。
