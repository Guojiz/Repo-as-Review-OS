# Goal Model

Repo as Review OS must be goal-driven.

The AI should not begin by dumping old material into the repository. It should first understand the user's goal, then decide what to keep, split, rewrite, generate, review, or archive.

## Why goals matter

A repository without a goal becomes an archive.

A repository with a goal becomes an operating system.

Examples of goals:

- prepare for an exam;
- improve math problem solving;
- build an essay material system;
- train vocabulary;
- prepare a research project;
- maintain a long-term learning record.

## Goal hierarchy

A goal should be stored in layers:

```text
Main goal
→ subgoals
→ weak points
→ source material
→ models
→ review tasks
→ automation outputs
```

## Example

If the main goal is exam preparation, the AI may create:

- subject goals;
- weak-topic lists;
- problem models;
- daily review plans;
- generated practice sets;
- progress records.

The AI should not only return old material. It should create new structure from old material.

## Goal file

Each Review OS can include:

```text
goals/main-goal.md
goals/subgoals.md
goals/progress.md
```

## AI behavior

Before organizing material, the AI should ask or infer:

1. What is the user's current main goal?
2. What is the time horizon?
3. What subjects or skills matter most?
4. What weak points are visible?
5. What should be generated next?
6. What should be archived or ignored?

## Innovation on top of old material

Old material is raw ore, not the final product.

The AI should transform it into:

- models;
- checklists;
- practice sets;
- dashboards;
- review cycles;
- reusable templates.

## Practical rule

Every file should answer one question:

**Which goal does this serve?**

If it serves no current or future goal, archive it or mark it as low priority.
