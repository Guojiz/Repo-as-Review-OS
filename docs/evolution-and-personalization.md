# Evolution and Personalization

Repo as Review OS is not a dump of everything the user has.

It is a living personal system. The AI should break material down, extract patterns, organize it, rewrite it, and remove what no longer helps.

## Not a dumping ground

Do not upload or copy everything into the repository.

The AI should process material through this pipeline:

```text
raw material
→ source record
→ extracted problem or idea
→ reusable model
→ review item
→ dashboard update
```

## Core actions

The AI should be able to:

- split large material into smaller useful units;
- summarize only after preserving the source status;
- extract common patterns;
- merge duplicate notes;
- mark weak or incomplete items;
- rewrite confusing files;
- delete or archive useless structure after confirmation;
- adapt the system to the user's habits.

## Personalization

There is no single fixed layout for every user.

A Review OS should adapt to:

- the user's subjects;
- exam or training goals;
- preferred language;
- available AI tool;
- permission level;
- local material situation;
- daily review rhythm.

## Nothing is sacred

Templates, folders, dashboards, and rules can all change.

The only stable principles are:

1. keep sources traceable;
2. separate source, summary, model, index, and todo;
3. keep local material protected;
4. use GitHub as the operating layer;
5. let the AI report what changed.

## AI behavior

The AI should not ask the user to design the whole system from scratch.

It should start with a small workable structure, then improve it as real material arrives.

## Practical rule

When in doubt, do not add more. Extract, merge, label, or simplify first.
