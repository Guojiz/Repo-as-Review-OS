# 运行模型

GitLearnOS 是用户拥有的轻量学习控制层。它不承包学习发生的场所，而是让一个可替换的主 Agent 自动整理分散输入、生成针对性问题并维护长期状态。

## 四层结构

```text
真实学习渠道
→ 老师、课堂、补习、纸笔、教材、题库、考试、同学、AI

一个主 AI Agent
→ 识别意图、整理、出题、执行、讲解、验证

GitLearnOS 规则、Skills 与 learning-policy
→ 控制来源、自动化、写回、安全和证据标准

目标 GitHub 仓库
→ 保存学习者拥有、可检查、可迁移、可撤销的状态
```

不需要按层或职责拆分多个 Agent。

## 三项核心能力

### 自动整理

```text
新输入
→ 标记渠道与来源
→ 关联目标和知识缺口
→ 去重与提炼
→ 更新处理状态
→ 给出下一步
```

### 针对性出题

```text
目标 + 来源/模型 + 知识缺口 + 历史表现 + 用户约束
→ 诊断题 / 练习题 / 变式题 / 迁移题 / 复习题 / 外部问题包
→ 答案或评分标准
→ 结果写回
```

### 自动执行

```text
用户事件或 Agent 接手
→ 读取 learning-policy
→ 执行安全、低风险、可逆操作
→ 原子写回与去重
→ 刷新 dashboard
→ 返回透明回执
```

后台定时执行只在当前平台有真实调度器时启用。

## 两个闭环

跨场景协调是主闭环：

```text
捕获
→ 关联
→ 路由给老师、学习者、AI 或其他渠道
→ 整理反馈或生成问题
→ 写回
→ 下一步
```

AI 教学是可选闭环：

```text
尝试
→ 诊断
→ 支持
→ 新题验证
→ 评分
→ 复习
```

## 状态模型

- `learning-policy.md`：自动化、隐私、写回与调度偏好；
- `dashboard.md`：当前优先级、到期任务、等待反馈与下一步；
- `learner-profile.md`：有长期证据支撑的学习者状态；
- `goals/`：目标与成功标准；
- `inbox/`：尚未完全处理的快速输入；
- `sources/`：老师、课堂、教材、平台、AI等来源与可用性；
- `models/`：跨来源整合的可复用理解；
- `knowledge-gaps/`：问题处理生命周期与掌握证据；
- `handoffs/`：给老师、同学或其他 Agent 的问题与反馈包；
- `reviews/`：生成题目、回答、评分与下次日期；
- `sessions/`：真正改变状态的重要跨渠道活动。

## 决策规则

创建或修改文件前判断：

```text
用户是否明确提供或授权？
是否服务于活跃目标？
是否已经有唯一状态拥有者？
未来是否仍有价值？
能否安全、低风险、可逆地自动执行？
```

如果只是临时内容，不写入。安全操作按 `safe-auto` 直接执行并汇报，高影响操作先询问。

## 外部解决与掌握

问题处理状态与掌握证据分离。老师已经讲清楚可以立即结束等待和重复教学；独立掌握仍根据需要通过作答、延迟回忆或迁移验证。用户可以选择不安排验证。

## 对标边界

DeepTutor 提供能力方向：来源支撑、动态学习者状态、个性化问题和反馈闭环。GitLearnOS 将这些能力压缩成跨渠道、用户拥有、由外部主 Agent 执行的文件和 Skills。

它有意不提供完整应用、服务端、多 Agent Runtime、RAG 引擎或数据库。
